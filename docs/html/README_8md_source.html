24 Building and solving a model
25 ----------------------------
26 
27 There are several \ref examples/finished "examples" and
28 \ref examples/tutorial "tutorials". These display some functionality of the
29 interface and can serve as an entry point for writing more complex code.
30 You might also want to have a look at this article about PySCIPOpt:
31 <https://opus4.kobv.de/opus4-zib/frontdoor/index/index/docId/6045>. The
32 following steps are always required when using the interface:
33 
34 1) It is necessary to import python-scip in your code. This is achieved
35  by including the line
36 
37 ``` {.sourceCode .python}
38 from pyscipopt import Model
39 ```
40 
41 2) Create a solver instance.
42 
43 ``` {.sourceCode .python}
44 model = Model("Example") # model name is optional
45 ```
46 
47 3) Access the methods in the `scip.pyx` file using the solver/model
48  instance `model`, e.g.:
49 
50 ``` {.sourceCode .python}
51 x = model.addVar("x")
52 y = model.addVar("y", vtype="INTEGER")
53 model.setObjective(x + y)
54 model.addCons(2*x - y*y >= 0)
55 model.optimize()
56 ```

58 Writing new plugins
59 -------------------
60 
61 The Python interface can be used to define custom plugins to extend the
62 functionality of SCIP. You may write a pricer, heuristic or even
63 constraint handler using pure Python code and SCIP can call their
64 methods using the callback system. Every available plugin has a base
65 class that you need to extend, overwriting the predefined but empty
66 callbacks. Please see `test_pricer.py` and `test_heur.py` for two simple
67 examples.
68 
69 Please notice that in most cases one needs to use a `dictionary` to
70 specify the return values needed by SCIP.
71 
72 Extending the interface
73 -----------------------
74 
75 PySCIPOpt already covers many of the SCIP callable library methods. You
76 may also extend it to increase the functionality of this interface. The
77 following will provide some directions on how this can be achieved:
78 
79 The two most important files in PySCIPOpt are the `scip.pxd` and
80 `scip.pyx`. These two files specify the public functions of SCIP that
81 can be accessed from your python code.
82 
83 To make PySCIPOpt aware of the public functions you would like to
84 access, you must add them to `scip.pxd`. There are two things that must
85 be done in order to properly add the functions:
86 
87 1) Ensure any `enum`s, `struct`s or SCIP variable types are included in
88  `scip.pxd` <br>
89 2) Add the prototype of the public function you wish to access to
90  `scip.pxd`
91 
92 After following the previous two steps, it is then possible to create
93 functions in python that reference the SCIP public functions included in
94 `scip.pxd`. This is achieved by modifying the `scip.pyx` file to add the
95 functionality you require.
96 
97 We are always happy to accept pull request containing patches or
98 extensions!
99 
100 Please have a look at our [contribution guidelines](CONTRIBUTING.md).

102 Gotchas
103 -------
104 
105 ### Ranged constraints
106 
107 While ranged constraints of the form
108 
109 ``` {.sourceCode .}
110 lhs <= expression <= rhs
111 ```
112 
113 are supported, the Python syntax for [chained
114 comparisons](https://docs.python.org/3.5/reference/expressions.html#comparisons)
115 can't be hijacked with operator overloading. Instead, parenthesis must
116 be used, e.g.,
117 
118 ``` {.sourceCode .}
119 lhs <= (expression <= rhs)
120 ```
121 
122 Alternatively, you may call `model.chgRhs(cons, newrhs)` or
123 `model.chgLhs(cons, newlhs)` after the single-sided constraint has been
124 created.
125 
126 ### Variable objects
127 
128 You can't use `Variable` objects as elements of `set`s or as keys of
129 `dict`s. They are not hashable and comparable. The issue is that
130 comparisons such as `x == y` will be interpreted as linear constraints,
131 since `Variable`s are also `Expr` objects.
132 
133 ### Dual values
134 
135 While PySCIPOpt supports access to the dual values of a solution, there
136 are some limitations involved:
137 
138 - Can only be used when presolving and propagation is disabled to
139  ensure that the LP solver - which is providing the dual
140  information - actually solves the unmodified problem.
141 - Heuristics should also be disabled to avoid that the problem is
142  solved before the LP solver is called.
143 - There should be no bound constraints, i.e., constraints with only
144  one variable. This can cause incorrect values as explained in
145  [\#136](https://github.com/SCIP-Interfaces/PySCIPOpt/issues/136)
146 
147 Therefore, you should use the following settings when trying to work
148 with dual information:
149 
150 ``` {.sourceCode .python}
151 model.setPresolve(pyscipopt.SCIP_PARAMSETTING.OFF)
152 model.setHeuristics(pyscipopt.SCIP_PARAMSETTING.OFF)
153 model.disablePropagation()
154 ```
