1 Contributing to PySCIPOpt
2 =========================
3 
4 Code contributions are very welcome and should comply to a few rules:
5 
6 0. Read Design principles of PySCIPOpt\_.
7 1. Compatibility with both Python-2 and Python-3.
8 2. All tests defined in the Continuous Integration setup need to pass:
9  - [.travis.yml](../../.travis.yml)
10  - [appveyor.yml](../../appveyor.yml)
11 
12 3. New features should be covered by tests *and* examples. Please
13  extend [tests](tests) and [examples](examples). Tests uses pytest
14  and examples are meant to be accessible for PySCIPOpt newcomers
15  (even advanced examples).
16 4. New code should be documented in the same style as the rest of
17  the code.
18 5. New code should be
19  [pep8-compliant](https://www.python.org/dev/peps/pep-0008/). Help
20  yourself with the [style guide
21  checker](https://pypi.org/project/pep8/).
22 6. Before implementing a new PySCIPOpt feature, check whether the
23  feature exists in SCIP. If so, implement it as a pure wrapper,
24  mimicking SCIP whenever possible. If the new feature does not exist
25  in SCIP but it is close to an existing one, consider if implementing
26  that way is substantially convenient (e.g. Pythonic). If it does
27  something completely different, you are welcome to pull your request
28  and discuss the implementation.
29 7. PySCIPOpt uses [semantic versioning](https://semver.org/). Version
30  number increase only happens on master and must be tagged to build a
31  new PyPI release.
32 
33 For general reference, we suggest:
34 
35 - [PySCIPOpt README](README.md);
36 - [SCIP documentation](http://scip.zib.de/doc/html/);
37 - [SCIP mailing list](https://listserv.zib.de/mailman/listinfo/scip/)
38  which can be easily searched with search engines (e.g.
39  [Google](http://www.google.com/#q=site:listserv.zib.de%2Fpipermail%2Fscip));

42 - [SCIP/PySCIPOpt Stack
43  Exchange](https://stackoverflow.com/questions/tagged/scip).
44 
45 If you find this contributing guide unclear, please open an issue! :)

47 Design principles of PySCIPOpt
48 ==============================
49 
50 PySCIPOpt is meant to be a fast-prototyping interface of the pure SCIP C
51 API. By design, we distinguish different functions in PySCIPOPT:
52 
53 - pure wrapping functions of SCIP;
54 - convenience functions.
55 
56 **PySCIPOpt wrappers of SCIP functions** should act:
57 
58 - with an expected behavior - and parameters, returns, attributes, ...
59  - as close to SCIP as possible
60 - without *"breaking"* Python and the purpose for what the language it
61  is meant.
62 
63 Ideally speaking, we want every SCIP function to be wrapped in
64 PySCIPOpt.
65 
66 **Convenience functions** are additional, non-detrimental features meant
67 to help prototyping the Python way. Since these functions are not in
68 SCIP, we wish to limit them to prevent difference in features between
69 SCIP and PySCIPOPT, which are always difficult to maintain. A few
70 convenience functions survive in PySCIPOpt when keeping them is
71 doubtless beneficial.
72 
73 Admittedly, *there is a middle ground where functions are not completely
74 wrappers or just convenient*. That is the case, for instance, of
75 fundamental `Model`{.sourceCode} methods like `addCons`{.sourceCode} or
76 `writeProblem`{.sourceCode}. We want to leave their development to
77 negotiation.
